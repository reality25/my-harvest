1) Add the missing indexes (biggest win)
Your triggers update public.posts / public.communities by PK (id), but your reads (feed, lists, joins, moderation, etc.) will likely filter by these columns. Add indexes aligned to common access patterns and to FK columns.

Recommended starting set:

SQL Query

2) Make the “counts” triggers correct under updates + safer under concurrency
Right now:

handle_reaction_count() handles UPDATE, but it does not address user_id changing (your PK prevents it, so likely fine).
Both count triggers do a read-modify-write pattern; concurrency can still be fine for simple +1/-1, but it’s safer/clearer to avoid counting drift by ensuring the logic is exactly per row transition (which you mostly do).
Readability improves if you split the “like/dislike” branching into one path.
Also: you’re using SECURITY DEFINER for triggers—good if you need to bypass RLS, but you should ensure the function owner is correct and that only the trigger can execute it (via function ownership/permissions).

Improved function body style (same logic, clearer):

SQL Query



create or replace function public.handle_reaction_count()
returns trigger
language plpgsql
security definer
as $$
begin
  -- INSERT
  if tg_op = 'INSERT' then
    if new.type = 'like' then
      update public.posts
      set like_count = like_count + 1
      where id = new.post_id;
    else
      update public.posts
      set dislike_count = dislike_count + 1
      where id = new.post_id;
    end if;
    return new;
  end if;

  -- DELETE
  if tg_op = 'DELETE' then
    if old.type = 'like' then
      update public.posts
      set like_count = greatest(like_count - 1, 0)
      where id = old.post_id;
    else
      update public.posts
      set dislike_count = greatest(dislike_count - 1, 0)
      where id = old.post_id;
    end if;
    return old;
  end if;

  -- UPDATE (handle type change)
  if old.type = new.type then
    return new;
  end if;

  if new.type = 'like' then
    -- moved to like
    update public.posts
    set like_count = like_count + 1,
        dislike_count = greatest(dislike_count - 1, 0)
    where id = new.post_id;
  else
    -- moved to dislike
    update public.posts
    set dislike_count = dislike_count + 1,
        like_count = greatest(like_count - 1, 0)
    where id = new.post_id;
  end if;

  return new;
end;
$$;

This has two advantages:

It avoids unnecessary updates when type didn’t change.
It’s easier to reason about.
Do the same for comment_count and member_count (return early by tg_op instead of nesting).

3) Ensure your RLS policies are valid for storage deletes
Your storage delete policy:

SQL Query



and auth.uid()::text = (storage.foldername(name))[1]

This assumes:

your object names are formatted so that the first “folder” equals the user id string (e.g. "<uid>/filename.jpg").
If your uploaded object paths don’t follow that convention, this policy will silently block deletes. For performance/readability, consider standardizing upload paths in your app and then using that same convention consistently.

Also, readability: give the policy names short + consistent (avoid quotes in policy names unless needed).

6) Drop “create policy if not exists” (important)
Postgres does not support CREATE POLICY IF NOT EXISTS—so this script will fail in a real migration unless Supabase preprocesses it (it doesn’t).

To make it idempotent, you need a different pattern, e.g.:

drop policy if exists ... then create policy ..., or
wrap with DO blocks.
Example pattern:

SQL Query



drop policy if exists "Public can view post media" on storage.objects;
create policy "Public can view post media"
  on storage.objects for select
  using (bucket_id = 'post-media');

This is both correct and more readable than pretending IF NOT EXISTS works.
