We already have a working Supabase-backed application with Google authentication, AI assistant, weather, news, and farm management features. Do NOT rebuild authentication or existing database logic. Your task is to design and implement a complete, production-ready user onboarding and profile system that integrates cleanly with the current architecture and enhances personalization across AI, weather, news, and alerts.

Ensure Google Auth is fully functional using Supabase Auth, including correct redirect URLs, session handling, and persistence. After first login, detect whether the user has completed onboarding. If not, immediately route them into a structured onboarding flow. This flow must be optional and skippable, but strongly encouraged through good UX, not forced friction. Users must be able to return later and complete or update their profile at any time.

The onboarding process must collect relevant personalization data for both farmers and non-farmers. Include fields such as full name, country, region/county, preferred language (if applicable), and role (farmer, agribusiness, student, researcher, or general user). For farmers, collect farm-related data such as farm type (crop, livestock, poultry, aquaculture, beekeeping, mixed farming, etc.), specific crops or animals, farm size, and primary activities. Provide a comprehensive predefined list of farming types and activities, but always include an “Other” option with free-text input so users are never blocked by missing categories.

Allow users to optionally upload a profile photo and optionally attach farm-related images. Support lightweight social features such as following other users or communities if those systems already exist in the schema. Do not invent disconnected features; integrate with existing tables like profiles, farms, farm_records, and communities where applicable.

All onboarding data must be stored properly in Supabase using the existing schema where possible. Extend the schema only if necessary, and do so cleanly without breaking existing relationships. Ensure the profiles table is the central source of truth for user identity and personalization fields such as country, region, farm scale, and role. Link farm-related inputs to farms, farm_entities, and farm_records where appropriate instead of duplicating data.

This personalization must directly power other systems. Weather services must use the user’s location (country/region or farm location) by default, with manual override still available. News must be filtered based on the user’s location and farming interests. AI assistant prompts must incorporate user profile context (farm type, crops, region, etc.) to produce more relevant advice. Alerts must be tailored using both weather data and the user’s farming profile (e.g., pest risks for specific crops, drought warnings for regions, livestock disease alerts where relevant).

The onboarding and profile UI must be clean, mobile-first, and responsive across phones, tablets, and desktops. Use a step-by-step flow with clear progress indicators, minimal friction, and the ability to skip or go back. Avoid overwhelming the user with too many inputs at once. Provide sensible defaults and pre-filled values where possible (e.g., detect country from IP or browser locale). Ensure all forms have proper validation, loading states, and error handling.

Profile editing must be accessible after onboarding through a dedicated settings or profile page. Changes must immediately reflect across the system (weather, news, AI, alerts) without requiring a reload or re-login. Ensure no hardcoded data is used anywhere; everything must come from real user input or live APIs.

Finally, audit the entire flow to ensure there are no broken links between authentication, onboarding, database storage, and downstream services. The goal is a seamless experience where a new user signs in with Google, optionally completes onboarding, and immediately receives personalized AI advice, weather insights, and agricultural news based on real data.
